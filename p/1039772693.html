2. File Types

  1. Regular File: 最常见的文件类型. Regular file包含数据, 无论二进制还是文本数据. 应用负责操作和解释数据.
  2. Directory File: 该文件可包含一些文件的名字或指向文件的指针. 进程可以读取directory file, 但只有kernel才能对directory file做写操作. 进程只能通过调用unix function来操作directory file.
  3. Block Special File: 一种提供buffered I/O访问(可读可写)的文件类型, 例如disk drive
  4. Character special file: 一种提供non-buffered I/O访问的文件格式, 例如terminal
  5. FIFO: 提供进程间的交流, 也被叫做pipe
  6. Socket: 提供进程间的网络形式交流.
  7. Symbolic link: 一种指向其他文件的指针

stat结构体的st_mode表示文件类型, 我们可以用过macro来判断某个文件是什么类型:

  1. S_ISREG(): regular file
  2. S_ISDIR(): directory file
  3. S_ISCHR(): character special file
  4. S_ISBLK(): block special file
  5. S_ISFIFO(): pipe or FIFO
  6. S_ISLNK(): symbolic link
  7. S_ISSOCK(): socket

3. User ID and Group ID

  1. real user ID: 当前进程的user id
  2. real group ID: 当前进程的group id
  3. effective user ID: kernel看待当前进程的user id
  4. effective group ID: kernel看待当前进程的group id
  5. saved set-user-ID: 进程提升用户权限时将原本的user id放入该域, 降低权限时使用
  6. saved set-group-ID: 进程提升组权限时将原本的group id放入该域, 降低权限时使用
  7. set-user-ID: 非文件owner用户执行文件时, 拥有文件owner的read/write/execute权限, 且可访问owner的其他文件
  8. set-group-ID: 非同group用户执行该文件夹时, 拥有文件夹owner的read/write/execute权限

当用户登录kernel时, read ID和effective ID都是同一个的ID. 当运行setuid时会将effective ID切换成其他ID(通常为root). 有时文件的拥有者并不想其他进程修改或读取文件内容, 但需要其他进程执行该文件, 这时可设置set-user-ID bit和set-group-ID bit为1, 让其他进程的effective ID变成文件拥有者的ID.

4. File Access Permissions

Unix中的每一个文件都有权限设置, 每个文件都有9个bits来表示文件权限, 分为三类:

st_mode_mask meaning
S_IRUSR user-read
S_IWUSR user-write
S_IXUSR user-execute
S_IRGRP group-read
S_IWGRP group-write
S_IXGRP group-execute
S_IROTH other-read
S_IWOTH other-write
S_IXOTH other-execute

其中文件夹的read表示进程可读取文件夹内所有filename, execute表示进程可进入文件夹. kernel以一套规则来判断当前进程是否有权限来对文件进行操作:

  1. 若effective user ID为0, 说明当前进程为superuser, 任何操作都被允许.
  2. 若当前进程的effective user ID与文件的user ID相同, 则可执行所有被允许的user access permission
  3. 若当前进程的effective group ID与文件的group ID相同, 则可执行所有被允许的gruop access permission
  4. 若other access permission有可执行的权限, 则可执行被允许的权限, 否则被denied

10. File Size

stat结构体中的st_size描述文件共有多少byte. 但st_size只对regular file, directory和symbolic link有效. FreeBSD 8.0, Mac OS X 10.6.8, and Solaris 10也定义了pipe的文件大小.
以下是一些特殊情况:

  1. 对于regular file, 文件大小可以为0, 文件的第一个字符即为end-of-file indication.
  2. 对于directory, 文件大小通常为8的倍数
  3. 对于symbolic link, 文件大小为文件名的byte数

绝大数UNIX系统都提供另外两个fields:

  1. st_blksize: block size for I/O
  2. st_blocks: number of 512 byte blocks

12. File System

UNIX支持多种file system. UFS就是基于Berkeley fast file system所构建的一种文件系统, 其中包含以下几个部分:

  1. boot block: 包含bootstrap code, 系统启动使用
  2. super block: 描述file system的状态:
  3. file system大小
  4. 文件数量
  5. free space的位置
  6. ownership
  7. cylinder group: 由磁盘的连续多个cylinder组成(无需移动磁头, 只需切换磁头), 其中包含两个最重要的部分:
  8. i-node group
  9. data blocks

Directory entry包含filename和指向inode的指针. 每个inode都有一个link count来统计有多少directory entry指向该inode(st_nlink, 数据类型为nlink_t, 这种link被称为hard link). 当st_nlink为0, 文件将被删除. inode包含大部分文件信息: 文件类型, 权限, 文件大小, 指向data blocks的指针和其他信息. 因此在不更改file system的情况下, 文件的内容不会被修改, 只会添加一个新的directory entry指向当前inode, 并unlink旧的directory entry.

15. Symbolic Links

Symbolic link(也称为soft link)是一种指向文件的间接指针. 与hard link不同, symbolic link不直接指向文件的i-node, 其克服了hard link的几个缺陷:

  1. hard link和其指向的file必须在一个文件系统中.
  2. 只有superuser才能创建指向文件夹的hard link.

当symbolic link产生一个路径循环时, 访问函数会返回一个ELOOP错误. 但symbolic link可以很轻易的unlink. 对于hard link就无法那么轻易地unlink, 因此系统不允许普通用户创建指向文件夹的hard link.

16. File Times

Field Description Example ls option
st_atim last-access time of file data read -u
st_mtim last-modification time of file data write default
st_ctim last-change time of i-node status chmod, chown -c
  1. st_ctim是文件的inode修改时间, 修改文件的权限, user ID, link数量等操作都会改变st_ctim
  2. access()和stat()只是访问inode, 而不是文件数据, 所以不会改变st_atim

21. Device Special Files

stat结构体中有两个成员变量: st_dev和st_rdev. 他们的类型都为dev_t(major和minor). major用于认定设备的驱动, minor用于确认其subdevice. 同一硬件上的文件系统拥有相同的major号码, 但不一定有同一minor号码. 只有character special file和block special file拥有st_rdev值, 这两个值主要用于判断两个filename或file descriptor是否位于同一device.
