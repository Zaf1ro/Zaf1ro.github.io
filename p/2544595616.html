Process Control

09/01/19

1. Process Indetifiers

每个进程都有一个唯一且非负的process ID. Process ID支持重用, 进程一旦被停用, 其process ID就可被其他新建的进程使用. 但大多数UNIX系统会推迟这一重用操作, 来保证新进程不会被当做刚被结束的进程而误用. 以下是特殊进程和其process ID:

  1. Process ID 0: 用于进程调度, 作为system process之一.
  2. Process ID 1: 用于bootstrap后的kernel初始化, 程序位于/etc/init或/sbin/init, 非system process却拥有superuser权限.
  3. Process ID 2: pagedaemon, 用于virtual memory的分页功能.

2. fork Function

Child process中fork()的返回值为0, 而不是parent process的PID, 因为child process可通过getppid()获得parent process的PID. 而单个进程可拥有多个child process, 所以parent process中fork()的返回值为child process的ID.

2.1 Memory Sharing

新创建的child process会复制其parent process的大部分内存数据, 包括parent process的data space, heap, stack. 所以child process和parent process并不共享内存中的数据, 除了text segment(read-only).
由于fork()会复制大量数据而降低运行速度, 且fork-exec model(在fork()生成的child process中调用exec()来运行其他程序)被大量使用, child process其实并没有使用到从parent process复制而来的数据, 所以copy-on-write应运而生. 新创建的child process一开始不会复制parent process的数据, 而是共享同一区域内存. 只有当某一进程(child process或parent process)试图写入数据时才会开始复制修改的区域.

2.2 File Sharing

除了内存数据, child process与parent process还共享file descriptor. 需要注意的是, parent process和child process共享file offset, 这就保证了child process可以在parent process所在的文件位置继续进行读写操作.

如果parent process和child process在没有同步机制的情况下同时对某一file descriptor进行操作, 很可能导致数据丢失或逻辑错误. 一般有两种方法来保证同步:

  1. Parent process等待child process对file descriptor操作完毕后再进行操作.
  2. Parent process和child process各自关闭不需要的file descriptor, 避免两个进程使用同一file descriptor

2.3 Other Sharing

以下是调用fork()后parent process和child process的其他共享部分:

  • Real user ID, real group ID, effective user ID, and effective group ID
  • Supplementary group IDs
  • Process group ID
  • Session ID
  • Controlling terminal
  • The set-user-ID and set-group-ID bits
  • Current working directory
  • Root directory
  • File mode creation mask
  • Signal mask and dispositions
  • The close-on-exec flag for any open file descriptors
  • Environment variables
  • Attached shared memory segments
  • Memory mappings
  • Resource limits

以下是parent process和child process不同的部分:

  • The return values from fork
  • The process IDs
  • Different parent process ID
  • The child’s tms_utime, tms_stime, tms_cutime, and tms_cstime values are 0
  • File locks set by the parent are not inherited by the child
  • Pending alarms are cleared for the child
  • The set of pending signals for the child is set to the empty set

3. vfork Function

vfork()与fork()有两点不同:

  1. vfork()生成的child process不复制parent process的地址空间, 所以共享parent process的所有数据. 只有调用exec()或exit()才会退出parent process的地址空间.
  2. fork()不保证child process和parent process的执行顺序, 而vfork()要求child process必须在parent process之前运行. parent process会被阻塞, 直到child process调用exec()或exit()退出进程.

4. exit Functions

五种正常终止进程的方式:

  1. main()中调用return(), 相当于调用exit()
  2. 调用exit(). 会调用所有atexit()注册的exit handle, 并关闭所有I/O streams. 由于exit()由ISO C定义, 所以不处理file descritptor, 多进程和任务控制.
  3. 调用_exit()或_Exit(). ISO C定义了_Exit()来终止进程且不调用exit handlers. _exit()由POSIX.1定义, 所以会关闭所有进程拥有的file descriptor, 并将该进程的所有子进程的归属权转移给init进程.
  4. 进程的最后一个线程的start routine中执行return(). 线程的返回值不会用作进程的返回值.
  5. 进程的最后一个线程调用pthread_exit().

三种异常的进程终止的方式:

  1. 调用abort(), 不会删除临时文件, 不关闭stream buffer, 也不会调用atexit()注册的函数
  2. 进程接收到特定的signal
  3. 进程的最后一个线程回应cancellation request

无论进程如何终止, kernel都会关闭进程的所有open descriptor, 并释放内存. 当进程被终止时, 其parent process会获得其exit status. 若进程调用exit function(exit, _exit, _Exit)终止进程, 则exit status为exit function参数; 若进程异常终止, kernel会生成一个termination status, 其parent process能通过wait()或waitpid()来获取该termination status.
若进程终止时, 其child process还在运行, 则其子进程的parent PID将改为1(init进程), 这样可保证所有进程在任何时刻都有一个valid parent PID.
若child parent终止, 但其parent process还在运行时, parent process可能还未调用wait()或waitpid()获取termination status. 因此kernel会将终止的进程信息保留在内存中, 其中包括: 被终止进程的PID, CPU运行时间, termination status. 这些已被终止但未被其parent process知晓的进程被称为zombie.
当某进程的parent process为init时, 该进程的终止不会导致其成为zombie. 因为无论进程何时终止, init都会调用wait()来获取其termination status.

5. wait and waitpid Functions

无论进程如何终止, kernel都会通过SIGCHLD信号来提醒其父进程. 其parent process可选择忽略该信号, 或提供signal handler来处理该信号. wait()或waitpid()可提供以下功能:

  • 若child process都在运行, 则阻塞parent process
  • 若child process被终止, 则wait()或waitpid()会立刻返回termination status给parent process
  • 若没有child process, 则立刻返回并报错

waitpid()与wait()的区别:

  1. waitpid()可等待指定PID的child process, wait()只能等待所有的child process
  2. waitpid()提供了nonblocking模式, 不需要阻塞parent process
  3. waitpid()提供了WUNTRACED和WCONTINUED来支持job control

8. exec Functions

当一个进程调用exec()后, 进程会被一个新的程序代替, 其PID不会改变, 但其text segment, data segment, heap, stack都会被替换成新的.

除了PID, exec后的进程还继承了原本的一些属性:

  • Process ID and parent process ID
  • Real user ID and real group ID
  • Supplementary group IDs
  • Process group ID
  • Session ID
  • Controlling terminal
  • Time left until alarm clock
  • Current working directory
  • Root directory
  • File mode creation mask
  • File locks
  • Process signal mask
  • Pending signals
  • Resource limits
  • Nice value
  • Values for tms_utime, tms_stime, tms_cutime, and tms_cstime

若exec()执行的文件中设置了set-user-ID bit, 则effective user ID会被设置为执行文件的owner ID; 若没有设置set-user-ID bit, 则effective user ID不变.
UNIX系统的实现中, 只有execve()在kernel中调用system call, 其他的exec函数只是library function, 最后通过调用execve()来调用system call

还有三种情况需要考虑:

  1. 若file descriptor设置了close-on-exec flag, 则exec会关闭该file descriptor
  2. exec会关闭所有directory streams, 因为opendir会为每个新建的directory stream设置一个close-on-exec flag
  3. exec会保留real user ID和real group ID. 若设置了set-user-ID bit, 则effective user ID设置为owner ID; 否则effective user ID不变. Effective group ID同理.

9. Change User IDs and Group IDs

UNIX系统中的文件权限基于User ID和Group ID. 当没有权限去访问的文件时, 需要修改进程的user ID或group ID. 设计程序时应遵循最少权限准则, 保证非必要用户无权限访问某些文件, 可防范一些安全风险.

以下是user ID被改变的情况:

9.3 Example

at()可以规定某个时间执行某个程序. 通过at()可了解saved set-user-ID的特性.

  1. 假设被执行的程序的owner为root, 且set-user-ID bit被开启, 则at()执行程序后的进程权限如下:
    real user ID = our user ID
    effective user ID = root (由于开启set-user-ID bit, 则EUID为owner ID)
    saved set-user-ID = root
  2. at()首先会调用seteuid()降低权限, 将EUID设置为RUID:
    real user ID = our user ID
    effective user ID = our user ID (changed)
    saved set-user-ID = root
  3. 当需要访问root控制的文件时, at()会提高权限, 将EUID设置为saved set-user-ID:
    real user ID = our user ID
    effective user ID = root (changed)
    saved set-user-ID = root
  4. 访问root的文件后, at()会调用seteuid()降低权限, 将EUID设置为RUID:
    real user ID = our user ID
    effective user ID = our user ID (changed)
    saved set-user-ID = root
  5. 若执行程序中途运行其他命令, 则会调用fork()生成子进程, 且子进程会调用setuid()修改所有ID:
    real user ID = our user ID (changed)
    effective user ID = our user ID (changed)
    saved set-user-ID = our user ID (changed)

11. system Function

ISO C定义了system()函数, 但函数的实现依赖于系统的实现.

需要注意的是, system()在设置了set-user-ID bit的程序里执行会造成安全漏洞, 因为system在执行fork和exec后, child process的effective user ID会变为执行程序的owner ID, 而不是parent process的effective user ID. set-group-ID同理.

12. Process Scheduling

UNIX提供nice value表示进程在kernel中的优先级. nice value越小优先级越大. 只有privileged process可降低nice value来提高优先级, 非privileged process只能增加nice value来降低优先级.

The Single UNIX Specification规定nice value的范围为[0, (2*NZERO)-1], NZERO由系统决定. 但没有规定fork()创造的child process如何继承nice value.

13. Process Times

任何进程都可调用times()函数来获取自身进程和其子进程的wall clock, user CPU time和system CPU time.
