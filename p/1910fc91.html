1. Signal Concepts

每个signal都有名字, 且以SIG开头. 不同的UNIX系统拥有的的signal数量也不同, 但基本都在[30, 40]区间. Signal定义在**<signal.h>**头文件中, 一个正整数表示一个signal, 0表示null signal, 作为一个特殊的signal存在.
Signal的生成有以下几点可能:

  • 键盘按下特定terminal key后, terminal driver生成signal. 例如: Control-C生成SIGINT signal
  • 硬件检测到异常发生, 例如: 除零, 无效的内存引用等. 硬件检测到异常后会通知kernel, kernel生成signal
  • 调用kill(2)函数向任意process或process group发出任意signal. 调用kill(2)的前提:
    1. kill(2)的调用者为superuser
    2. kill(2)的调用者为signal接收进程的owner
  • 执行kill(1) command, 功能与kill(2)函数相同, 只是作为kill(2)的command接口
  • 软件层面检测到异常并发出signal, 例如: SIGURG, SIGPIPE, SIGALRM等

Signal作为异步事件之一, 进程无法直接读取某个变量判断是否接收到signal. 进程只需告诉kernel, 假如向该进程发出signal, 进程该做什么. 以下是signal接收时进程可选择的操作:

  1. 无视signal: 除了SIGKILL和SIGSTOP, 其他signal都可被无视. 注意: 无视硬件异常发出的signal会导致undefined behavior.
  2. 捕获signal: 想要捕获signal就要传给kernel一个函数, 一个用于处理该signal的函数. 捕获signal通常是为处理该进程之外突发的事件, 例如: 进程接收到SIGCHLD, 说明其child process已经执行完毕, 这时应调用waitpid()获取child process的termination status.
  3. 默认操作: 每个signal都有一个默认操作. 若进程没有指定捕获signal的函数, 则signal接收后按照默认操作执行.

4. Interrupted System Calls

system call分为两种: slow system call和fast system call. slow system call会一直阻塞调用进程, 直到完成操作, 包含以下几类:

  • 读取某些文件类型(pipes, terminal device, network device)
  • 写入某些文件类型(buffer溢出或其他原因)
  • 打开某些需要条件触发的文件(例如: modem响应时才可打开terminal device)
  • pause()和wait()函数
  • 某些ioctl操作
  • 某些IPC函数

5. Reentrant Functions

当进程收到signal, 进程会暂时停止运行并执行signal handler中的命令. Signal handler运行结束后会继续执行进程中未完成的命令. Signal handler无法得知当前进程执行到了哪一步, 所以执行某些命令可能会导致异常. 例如: 进程在运行malloc()过程中接收到signal并将执行权转移给signal handler, signal handler中调用malloc()会导致进程中malloc()设置的全局变量被修改, 从而导致无法预测的异常.

  1. 使用了static data structure
  2. 调用malloc()或free()
  3. standard I/O library

需要注意: signal handler中调用的reentrant function可能产生新的errno, 并覆盖进程原本产生的errno. 所以调用reentrant function之前应备份errno, 并在退出signal handler前还原errno. longjmp()同样不推荐在signal handler中使用, longjmp()为non-asynchronous-safe function(SIG30-C), 会导致未定义行为.

6. SIFCLD Semantics

SIGCHLD signal与其他signal相同. 当child process状态发生变化是, parent process会收到该signal, parent process可通过调用wait来获取termination status.
早期UNIX系统处理SIGCLD的方式则不同:

  1. SIG_DFL: SIGCLD的默认动作为无视该signal, 但也不会丢弃该signal, 所以还是可以用wait来获取child parent的信息, 否则child process变为zombie.
  2. SIG_IGN: 如果调用signal()将SIGCLD设置为SIG_IGN, 则child process的信息会被无视并丢弃. 使用wait会阻塞所有child process并返回ECHILD, child process不会变为zombie.
  3. 自定义处理方式: kernel会在signal()执行后立即检查是否存在child process状态改变.

7. Reliable-Signal Terminology and Semantics

之前提到, 早期UNIX的signal是不可靠的. 现代UNIX系统不存在signal handler被调用后重置为默认动作的情况, 所以不需要每次调用signal(); 提供block signal的方法, 当进程block某个signal后, 该signal在被触发后不会直接传给进程, 而是悬停, 直到进程不再block该signal. sigpending()可查看所有blocked signal.
若进程unblock某signal前已经有多个该signal被触发, 则结果视系统实现而定, POSIX.1并没有明确规定. 若系统创建了queue来存放每个signal, 则进程会收到多个signal; 但大部分UNIX系统不会创建queue保存多个signal, 只会保留一个. 进程unblock signal后, 多个signal抵达进程的顺序视系统实现而定, POSIX.1也没有明确规定.
每个进程都有一个signal mask来保存哪些signal被block. sigprocmask()可修改或查看mask. signal mask中每一bit都代表一个signal, 由于部分UNIX系统中的signal数量超过整数类型的bit个数, 所以POSIX.1规定了sigset_t来表示signal set.

16. abort Function

ISO C没有规定abort()是否应flush output stream或删除temporary files. POSIX.1允许abort()在进程终止前调用fclose()来关闭standard I/O streams.

17. system Function

POSIX.1要求system()忽略SIGINT, SIGQUIT, 并block SIGCHLD.
之所以忽略SIGINT和SIGQUIT, 因为system()的调用者其实不是该函数的执行主体, system()中fork()产生的child process才是运行主体. SIGINT和SIGQUIT的触发会让parent process和child process都收到signal, 而实际上只需要child process收到即可.
之所以block SIGCHLD, 是为了避免parent process过早收到SIGCHLD

20. Job Control Signals

POSIX.1规定了以下6个signal作为job control signals:

  • SIGCHLD: child process has stopped or terminated
  • SIGCONT: continue process if stopped
  • SIGSTOP: stop signal(cannot be caught or ignored)
  • SIGTSTP: interactive stop signal
  • SIGTTIN: read from controlling terminal by a background process group memeber
  • SIGTTOU: write to controlling terminal by a background process group member

除了SIGCHLD, 大部分进程都不会接触到剩下的5个signal, 因为shell会与这5个signal打交道: 当按下suspend character时, SIGTSTP会发送给foreground process group中的所有进程; 当想要恢复某个job时, SIGCONT会发送给指定process group中的所有进程; 当backgroup process group中的进程想要从shell中读取或写入数据时, 会收到SIGTTIN或SIGTTOU.
Job control signals之间也存在交互: 当stop signals(SIGTSTP, SIGSTOP, SIGTTIN, SIGTTOU)中的某一个发送给进程, 被阻塞的SIGCONT会被直接丢弃; 当SIGCONT被发送至进程时, 所有被阻塞的stop signal也会被丢弃.
当进程处于stopped状态时, SIGCONT会让进程恢复运行; 但若进程处于运行状态, 则SIGCONT会被忽略.
