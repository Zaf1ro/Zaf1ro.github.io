Standard I/O

08/20/19

1. Stream and FILE Objects

前两篇主要聚焦在File I/O. 本篇则开始讨论Standard I/O. File I/O偏向于系统层面的I/O实现, 一般为系统调用, 不具有系统可移植性, 且操作的对象为file descriptor. 而Standard I/O偏向于标准化模型, 具有一定的移植性. Standard I/O还包含一些优化, 例如buffer, 优化I/O执行长度等, 操作的对象为stream. 当使用standard I/O打开文件时, 会关联该文件到一个stream.
在ASCII字符集中, 单个字符由一个byte组成; 但在国际字符集中, 单个字符由多个byte组成. Standard I/O通过stream orientation来判断字符由单个或多个byte组成. Stream被创建时并没有orientation, 但如果单个或多byte的函数操作过该stream, 则该stream的orientation就被定为单个或多个byte. freopen()和fwide()可更改stream orientation.

2. Buffer

有三个stream不用进程创建即可使用: standard input, standard output和standard error. 这三个stream分别对应三个file descriptor: STDIN_FILENO, STDOUT_FILENO, STDERR_FILENO.
Standard I/O自动为读写操作提供了buffer, 使得磁盘读取次数减少, 并分离了应用程空间与实际的物理设备. Buffer分为以下三种:

  1. Fully buffered: 当buffer用完, 进程调用fflush()或进程退出时自动I/O操作.
  2. Line buffered: 当buffer用完, 遇到newline, 进程调用fflush或进程退出时进行I/O操作.
  3. Unbuffered: 不缓存任何数据, 每次调用I/O函数都会进行I/O操作.

ISO C的buffer具有以下两种属性:

  1. Standard input和output都必须使用fully buffered, 除非他们对应的不是interactive device.
  2. Standard error永远不使用fully buffered.

所以并没有明确规定standard error使用line buffered或unbuffered. 大部分情况下standard error使用unbuffered; Standard input和ouput stream在面对terminal device时使用line buffered; 其他情况下使用fully buffered. 也可以调用以下函数自定义buffer:

若进程以可读写的方式打开文件, 由于FILE stream使用buffer来缓存数据, 而input与output的切换会导致buffer内容被清空. 为避免数据丢失, 需遵守以下两条规则:

  1. 在output操作(例如: fwrite())后若需进行input操作(例如: fread()), 必须在两个操作间运行fflush, fseek, fsetpos或rewind
  2. 在input操作后若需进行output操作, 必须在两个操作间运行fseek, fsetpos, rewind. 若input操作已经达到EOF, 则不需要该步骤.

4. Read and Write a Stream

打开stream后有三种读写方式:

  1. Character-at-a-time I/O: 使用fgetc()或fputc()读取或写入一个字符.

  3. Direct I/O: 使用fread()或fwrite()读取或写入固定长度的字符.

6. Position a Stream

从standard I/O stream version 7开始提供了ftell()和fseek()来设置文件位置.

但若文件过大, 文件大小超过了long的最大范围, 则ftell()和fseek()无法正确处理. 所以应避免使用ftell()和fseek(), 而使用ftello()和fseeko(). ftello()和fseeko()功能上与ftell(), fseek()完全相同, 但使用off_t替代long来处理更大文件.

但对于含有不同编码的文件来说, ftello()和fseeko()并不能很好处理. ftello()和fseeko()只适合处理ASCII编码的文件, 也就是单个byte长度的字符. 如果处理国际字符编码的文件, 需使用fgetpos()和fsetpos().

9. Memory Streams

Standard I/O library会将数据缓存在buffer来加快处理速度, 且进程可创建自己的buffer. Single UNIX Specification Version 4后提供了memory streams来避免使用底层存储设备, 对于FILE stream的所有操作都在内存发生. 从而更快的处理数据.

相对于file-based standard I/O streams, memrory stream有几点不同:

  1. Memory stream的type设置为append后, file position会设置为buffer中的第一个NULL byte的位置. 若buffer中没有NULL byte, 则以buffer最后一个字节为file position.
  2. 若buf为NULL, 则type不应设为只读或只写. 因为fmemopen()得到的stream无法获取buffer的地址, 所以写入的数据永远无法读取. 同理, buffer读取的内容永远无法修改或添加.
  3. 当调用fclose(), fflush(), fseek(), fseeko()或fsetpos()时会在当前file position处添加一个NULL byte.
